# 招牌对比与报告模块分析

## 一、模块概述

招牌对比与报告模块是"街角招牌考古册"应用的核心分析功能模块，包含三个主要页面组件：

| 页面组件 | 路由路径 | 功能定位 | 最大数量 |
|---------|---------|---------|---------|
| `Compare.tsx` | `/compare` | 招牌对比分析主页面，提供多维度详细对比 | 4 块 |
| `CompareArea.tsx` | `/exhibition/compare` | 展厅内展品对比区，按主题分类的对比体验 | 4 块 |
| `CompareReport.tsx` | `/compare-report` | 收藏对比报告，生成简版对比结论 | 6 块 |

### 2.1 状态架构图

### 2.2 核心状态来源详解

#### 2.2.1 SignboardsContext — 招牌数据源

**文件**: `src/context/SignboardsContext.tsx`

- **状态存储**: `signboards: Signboard[]`
- **初始化**: 优先从 `localStorage['signboard-data']` 读取，降级使用 `data/signboards.ts` 默认数据
- **持久化**: `useEffect` 监听 `signboards` 变化，自动写入 localStorage
- **元数据**: 同时管理 tags、eras、fontStyles、colorPresets 等元数据，存储于 `localStorage['signboard-metadata']`

**关键方法**:
- `getSignboard(id)` — 按ID查询单块招牌
- `addSignboard(data)` / `updateSignboard(id, updates)` / `deleteSignboard(id)` — 增删改

#### 2.2.2 FavoritesContext — 对比/报告列表状态

**文件**: `src/context/FavoritesContext.tsx`

三个独立的ID列表，全部使用 localStorage 持久化：

| 状态变量 | localStorage Key | 最大容量 | 说明 |
|---------|-----------------|---------|------|
| `favorites` | `signboard-favorites` | 无限制 | 收藏的招牌ID数组 |
| `compareList` | `signboard-compare` | 4 (`MAX_COMPARE`) | 对比列表ID数组 |
| `reportList` | `signboard-report` | 6 (`MAX_REPORT`) | 报告列表ID数组 |

**核心方法模式**:

```typescript
// 添加/移除（切换）
toggleCompare(id: string): void
toggleReport(id: string): void

// 受控添加（带返回结果）
addToCompare(id: string): { success: boolean; reason?: string; alreadyIn: boolean }

// 批量操作
clearCompare(): void
clearReport(): void
setReportList(ids: string[]): void

// 状态查询
isInCompare(id: string): boolean
isInReport(id: string): boolean

// 数据过滤 — 将ID列表映射为完整招牌对象
getCompareSignboards(all: Signboard[]): Signboard[]
getReportSignboards(all: Signboard[]): Signboard[]
```

> **设计要点**: 上下文只存ID，不存完整对象。页面组件需要同时消费 `useSignboards()` 和 `useFavorites()`，通过过滤函数得到完整数据。这种"ID引用"模式保证了数据一致性——招牌数据修改后，所有引用自动更新。

### 2.3 数据流方向

```
用户操作 (添加对比)
    │
    ▼
SignboardCard.onClick → addToCompare(id)
    │
    ▼
FavoritesContext.compareList 更新
    │
    ├─→ localStorage 持久化 (自动)
    │
    └─→ 所有消费该 context 的组件重渲染
         │
         ├─→ Navbar 徽章数字更新
         ├─→ Compare 页面表格更新
         ├─→ CompareArea 页面更新
         └─→ SignboardCard 按钮状态更新
```

## 三、页面联动逻辑

### 3.1 入口分布

对比功能的入口分布在应用的多个位置，形成网状联动：

#### 3.1.1 导航栏入口 (`Navbar.tsx`)

- 路径: `src/components/Navbar.tsx:81-90`
- 直接导航到 `/compare` 和 `/compare-report`
- 显示角标徽章（badge）展示当前已选数量

#### 3.1.2 招牌卡片入口 (`SignboardCard.tsx`)

- 路径: `src/components/SignboardCard.tsx:177-196`
- 每张招牌卡片都有"对比"按钮
- 点击时调用 `addToCompare()` 或 `toggleCompare()`
- 成功/失败通过 Toast 提示用户
- 按钮状态随 `isInCompare(id)` 实时变化

#### 3.1.3 首页入口

- 路径: `src/pages/Home.tsx` (通过 SignboardCard 组件)
- 所有招牌卡片均可直接加入对比

#### 3.1.4 展厅入口 (`ExhibitionHall.tsx`)

- 路径: `src/pages/ExhibitionHall.tsx:254-263`
- "全部加入对比"按钮 — 批量添加当前主题下的所有招牌
- "前往对比区"按钮 — 跳转到 `/exhibition/compare`
- 单张展品卡片也可加入对比（通过 SignboardCard）

#### 3.1.5 收藏页入口 (`Favorites.tsx`)

- 路径: `src/pages/Favorites.tsx:298-302`
- "生成对比报告"按钮 — 跳转到 `/compare-report`
- 收藏的招牌卡片也可加入对比

#### 3.1.6 对比页内部跳转

- `Compare.tsx` 提供 "添加招牌 →" 链接跳回首页
- `CompareArea.tsx` 提供 "← 返回展厅" 链接
- `CompareReport.tsx` 提供 "← 返回收藏" 链接

### 3.2 页面间状态共享

由于 `FavoritesContext` 是全局 Provider（包裹在 `App.tsx` 最外层），三个页面共享同一份状态：

```
App.tsx
  └─ ThemeProvider
     └─ SignboardsProvider
        └─ FavoritesProvider    ← 全局唯一实例
           └─ ...
              ├─ /compare       ← 共享 compareList
              ├─ /exhibition/compare  ← 共享 compareList
              └─ /compare-report     ← 共享 reportList
```

**关键特性**:
- 在首页添加的招牌，切换到对比页立即显示
- 在展厅对比区添加的招牌，主对比页也能看到
- 对比列表和报告列表相互独立，互不干扰

### 3.3 对比页面内部联动

#### 3.3.1 Compare.tsx 主对比页

**表格联动**:
- 顶部表头显示招牌缩略图、名称、移除按钮
- 点击移除按钮 → `toggleCompare(id)` → 表格实时更新
- 点击招牌名称 → 跳转详情页 `/signboard/:id`

**分析模块联动**:
- 颜色分析、字体分析、年代分析三大模块
- 全部基于 `compareSignboards` 派生计算
- 对比列表变化时，所有分析数据自动重新计算

#### 3.3.2 CompareArea.tsx 展厅对比区

**Tab切换**:
- 总览对比、书法对比、色彩对比、历史对比 四个标签页
- 使用 `useState<'overview' | 'calligraphy' | 'color' | 'history'>` 管理
- 切换 Tab 不影响对比列表数据

**侧边栏联动**:
- 选择主题展厅 → 显示该主题下的招牌列表
- 点击列表项 → 调用 `handleAddFromTheme()` → `addToCompare(id)`
- 已添加的项显示"已添加"状态，不可重复点击

#### 3.3.3 CompareReport.tsx 对比报告

**两阶段流程**:
```
select 阶段 (选择招牌)
    │
    │ 点击"生成对比报告"
    │
    ▼
result 阶段 (展示报告)
    │
    │ 点击"重新选择" / "打印报告"
    └─→ 返回 select 阶段 / 调用 window.print()
```

**选择阶段**:
- 数据源: 仅从收藏夹中选择（`getFavoriteSignboards()`）
- 支持全选前 N 个 (`handleSelectAll()`)
- 支持清空选择
- 至少选择 2 块才能生成报告

**结果阶段**:
- 概览卡片、字体差异、颜色差异、年代差异四大部分
- 所有分析数据通过 `useMemo` 派生:
  - `colorAnalysis` — 颜色分析结果
  - `fontAnalysis` — 字体分析结果
  - `eraAnalysis` — 年代分析结果

## 四、派生计算逻辑

三个页面包含大量的派生计算逻辑，纯函数计算，无副作用。

### 4.1 颜色分析逻辑

| 函数 | 位置 | 功能 |
|------|------|------|
| `hexToRgb(hex)` | 三处重复定义 | HEX转RGB对象 |
| `getColorCategory(hex)` | Compare / CompareReport | 颜色色系分类（白色系、红色系等10类） |
| `getColorWarmth(hex)` | 三处重复定义 | 冷暖色调判断（warm / cool / neutral） |
| `getColorRatios(colors)` | Compare.tsx | 计算颜色占比（主色/辅色/点缀色分配比例） |
| `getColorRole(index, total)` | Compare.tsx | 颜色角色命名（主色、辅色、点缀色） |

**颜色占比算法** (`getColorRatios`):
- 1种颜色 → [100%]
- 2种颜色 → [60%, 40%]
- 3种颜色 → [55%, 30%, 15%]
- 4种及以上 → [50%, 28%, ...均分剩下的]
- > ⚠️ 注意：这是估算值，不是真实图像分析结果

### 4.2 字体特征分析

- 数据来源: `fontStyleFeatures` 静态映射表（7种字体风格）
- 包含: 笔画特征、整体气质、流行年代、特征标签列表
- CompareArea 和 CompareReport 中均有重复定义

### 4.3 年代差异分析

| 函数 | 位置 | 功能 |
|------|------|------|
| `getEraStageByYear(year)` | `types/index.ts` | 根据年份返回历史阶段 |
| `getEraDiff(year1, year2)` | Compare / CompareReport | 计算年代差异等级 |

**年代差异等级**:
| 差值范围 | 等级 |
|---------|------|
| ≤ 10年 | 同期 |
| 11-30年 | 相近 |
| 31-60年 | 跨代 |
| 61-100年 | 跨世纪 |
| > 100年 | 跨越百年 |

### 4.4 修缮统计

- `eventTypeCount(signboard, type)` — 统计某类修缮事件数量
- 类型: `restoration`(修缮) / `damaged`(受损) / `repaint`(重漆) 等

## 五、潜在边界风险

### 5.1 数据一致性风险

#### 🔴 风险1: 孤儿ID问题

**问题描述**:
`compareList` / `reportList` / `favorites` 只存储招牌ID。当招牌被删除（通过图鉴编辑台）时，这些列表中可能残留无效的"孤儿ID"。

**影响**:
- `getCompareSignboards()` 用 `filter` 过滤，孤儿ID会被静默忽略
- 表现为对比列表"少了一块"但用户不知道为什么
- Navbar 上的徽章数字与实际展示数量不一致

**相关代码**:
- `FavoritesContext.tsx:108-115` — 过滤方法只做简单 `includes` 匹配
- `SignboardsContext.tsx:166-168` — 删除招牌时不清理关联引用

**建议**: 删除招牌时同步清理 favorites、compareList、reportList 中的对应ID。

---

#### 🟡 风险2: localStorage 数据损坏

**问题描述**:
所有状态都依赖 localStorage 持久化。如果存储数据被手动篡改、或JSON格式损坏，会导致解析失败。

**当前处理**:
- SignboardsContext 有 try-catch，失败时降级为默认数据 (`SignboardsContext.tsx:68-73`)
- FavoritesContext **没有** try-catch 保护 (`FavoritesContext.tsx:32-45`)

**影响**:
- 若 `signboard-compare` 存储了非法JSON，`JSON.parse` 会抛出异常，导致整个应用白屏

**建议**: FavoritesContext 初始化时也应添加 try-catch 容错。

---

### 5.2 边界值处理风险

#### 🟡 风险3: 空列表边界

**问题描述**:
三个页面都处理了"空状态"（无对比项时的占位提示），但部分计算逻辑未做空判断保护。

**已处理**:
- Compare.tsx: 有 `compareSignboards.length === 0` 判断，空状态不渲染分析模块
- CompareReport.tsx: `useMemo` 中有 `reportSignboards.length < 2` 返回 null 的判断

**潜在问题**:
- 部分 reduce 操作可能因空数组而出错（如 Math.min(...[]) = Infinity）
- `compareSignboards.reduce((oldest, s) => ...)` 在空数组时会抛异常

**当前情况**: 空列表时整个分析区不渲染，因此 reduce 不会执行。但如果未来代码调整，可能引入bug。

---

#### 🟡 风险4: 单元素对比

**问题描述**:
对比列表只有1块招牌时，"对比"失去意义。共用颜色、年代差等需要两两比较的分析会显示异常值。

**当前表现**:
- Compare 页面：即使只有1块也显示完整分析面板
- "共用颜色"会显示全部颜色（因为只有一个集合，交集就是自身）
- "年代跨度"为0年

**建议**: 少于2块时，禁用部分对比分析或给出提示。

---

#### 🟡 风险5: 最大容量边界

**问题描述**:
`toggleCompare` 超过最大数量时**静默忽略**，不提示用户。

```typescript
// FavoritesContext.tsx:65-75
const toggleCompare = (id: string) => {
  setCompareList(prev => {
    if (prev.includes(id)) return prev.filter(c => c !== id);
    if (prev.length >= MAX_COMPARE) return prev;  // 静默失败
    return [...prev, id];
  });
};
```

**影响**:
- SignboardCard 中使用的是 `addToCompare`（有返回值和Toast），没问题
- 但如果其他地方直接调用 `toggleCompare`，满了之后用户点击无反馈

**建议**: 统一使用 `addToCompare` 模式，或给 toggle 也增加回调/提示机制。

---

### 5.3 计算逻辑风险

#### 🟡 风险6: 颜色占比估算不准确

**问题描述**:
`getColorRatios()` 使用固定比例分配，不是真实的图像颜色占比分析。

**影响**:
- 用户可能误以为是精确分析结果
- 颜色顺序变化（如数组重排）会导致占比结果完全不同
- 与"主色/辅色"的角色描述强绑定，顺序敏感

**建议**: 页面上增加"颜色占比为估算值"的提示说明。

---

#### 🟡 风险7: 冷暖色判断阈值

**问题描述**:
`getColorWarmth()` 中 `Math.abs(warmScore - coolScore) < 60` 判为中性，60是经验值，可能导致部分颜色分类不符合直觉。

**影响**: 分类结果可能与用户感知有偏差。

---

#### 🟡 风险8: 修缮历史空数组

**问题描述**:
`eventTypeCount` 对空数组返回0，没问题。但：

```typescript
// Compare.tsx:262-269
const years = s.restorationHistory.map(h => h.year);
const min = Math.min(...years);  // 空数组 → Infinity
const max = Math.max(...years);  // 空数组 → -Infinity
```

**影响**:
- 没有修缮历史的招牌，"历史跨度"显示 `Infinity - -Infinity`
- 目前数据中所有招牌都有修缮历史，所以没触发
- 如果用户新增招牌时未填写修缮记录，会出现异常

**建议**: 增加空数组判断，提供默认值或显示"暂无记录"。

---

### 5.4 代码维护风险

#### 🟠 风险9: 大量重复代码

**问题描述**:
以下逻辑在三个页面中重复定义，违反DRY原则：

| 重复项 | Compare | CompareArea | CompareReport |
|-------|---------|-------------|---------------|
| `conditionLabels` | ✅ | ✅ | ❌ |
| `fontStyleFeatures` | ✅ | ✅ | ✅ |
| `hexToRgb()` | ✅ | ✅ | ✅ |
| `getColorCategory()` | ✅ | ❌ | ✅ |
| `getColorWarmth()` | ✅ | ✅ | ✅ |
| `eventTypeCount()` | ✅ | ✅ | ❌ |
| `getEraDiff()` | ✅ | ❌ | ✅ |

**影响**:
- 修改一处容易忘记改别处，导致不一致
- 增加维护成本
- 测试覆盖困难

**建议**: 提取到 `utils/` 目录下的工具模块中统一管理。

---

#### 🟠 风险10: 报告列表与收藏列表耦合过强

**问题描述**:
CompareReport 的数据来源**仅限收藏夹**，未收藏的招牌不能加入报告。

```typescript
// CompareReport.tsx:71
const favoriteSignboards = getFavoriteSignboards(signboards);
// 选择阶段只渲染 favoriteSignboards
```

**影响**:
- 用户想对比但不想收藏时，无法生成报告
- 对比列表和报告列表完全隔离，无法互通

**建议**: 考虑允许从对比列表直接生成报告，或提供"将对比项加入报告"功能。

---

### 5.5 状态同步风险

#### 🟡 风险11: 多标签页数据不同步

**问题描述**:
localStorage 是持久化的，但 React state 只在当前标签页维护。用户打开多个标签页时：
- 标签页A添加了对比项
- 标签页B看不到变化（除非刷新页面）

**影响**: 轻度问题，多标签页使用场景较少。

**建议**: 可监听 `storage` 事件同步跨标签页状态。

---

### 5.6 打印报告风险

#### 🟡 风险12: 打印样式未优化

**问题描述**:
CompareReport 提供了"打印报告"按钮（`window.print()`），但可能没有专门的打印样式。

**影响**:
- 打印效果可能不理想（背景色丢失、布局错乱）
- 导航栏、页脚等无关内容也会被打印

**建议**: 添加 `@media print` CSS规则优化打印体验。

---

## 六、总结

### 6.1 架构优点

1. **状态分离清晰**: 招牌数据与收藏/对比状态分层管理，职责明确
2. **ID引用模式**: 上下文只存ID，保证数据一致性
3. **全局共享**: 单一 FavoritesContext 实例，多页面状态同步
4. **持久化完善**: 所有用户状态自动保存到 localStorage
5. **派生计算使用 useMemo**: 性能优化意识良好

### 6.2 核心改进建议（按优先级）

| 优先级 | 建议 | 影响 |
|-------|------|------|
| 🔴 高 | 修复孤儿ID问题：删除招牌时同步清理关联列表 | 数据一致性 |
| 🔴 高 | FavoritesContext 初始化增加 try-catch | 稳定性 |
| 🟠 中 | 提取重复工具函数到 utils 模块 | 可维护性 |
